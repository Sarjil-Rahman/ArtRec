# Data card

## Dataset summary

The project uses synthetic data generated for this repository. It represents a small art marketplace with artworks, users, impressions, interactions, and conversions.

The data is committed because it is small and lets the API run immediately after cloning the repo.

## Main files

- `data/raw/catalog.csv`: artwork metadata such as style, medium, artist, price, margin, tags, and descriptions.
- `data/raw/users.csv`: synthetic user preferences and price sensitivity.
- `data/raw/impressions.csv`: artworks shown to users in simulated sessions.
- `data/raw/interactions.csv`: clicks, saves, carts, purchases, and not-interested events.
- `data/raw/conversions.csv`: immediate and delayed synthetic purchases.
- `data/processed/train_features.csv`: feature rows used for training.
- `data/processed/validation_features.csv`: later feature rows used for validation.
- `data/processed/test_features.csv`: held-out feature rows.

## How the data is generated

The simulator creates a catalogue and users with latent preferences. It then creates sessions and interactions using hand-written behaviour rules. These rules make the data useful for testing a recommendation pipeline, but they are not a replacement for real marketplace behaviour.

## Known limitations

- The data is not from real customers.
- There is no demographic or protected attribute data.
- Prices, margins, and preferences are generated by rules.
- Real browsing noise, seasonality, stock issues, returns, fraud, and marketing campaigns are not fully represented.
- Offline results can look cleaner than they would with production traffic.

## Why synthetic data is still useful here

Synthetic data lets the full pipeline be reviewed without privacy risk. It also makes the repo easy to run. For a real deployment, I would replace it with consented production logs and keep the same checks around leakage, validation, reporting, and monitoring.
